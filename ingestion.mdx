Actioneer CDP is a managed, first-party event warehouse. Send your product events over one JSON-over-HTTPS API and Actioneer stores them in a warehouse it runs for you: no storage to provision, no pipeline to keep alive. The data stays first-party and owned by you, carries a full audit trail, and is queryable in natural language with the rest of your data.

It does what a hosted product-analytics tool does, capturing events from your apps and websites, but keeps that data in a warehouse you own and govern rather than in a separate analytics silo.

## Two ways to get data in

Actioneer takes data two ways. The CDP is the managed-warehouse path, and this page covers it.

- **Query in place.** Connect an existing warehouse, lake, or BI source and Actioneer reads it read-only, where it lives. No copy, no migration. See [Integrations](/integrations/data-warehouses).
- **Managed warehouse (this API).** Push events from your app and Actioneer stores them in a first-party warehouse it runs. Use this when you emit product events and don't want to run analytics storage yourself, or when you're moving off a hosted analytics tool and want that data back in a warehouse you own.

If your app already emits analytics events, you forward them straight to Actioneer. A single envelope carries event properties, profile updates, and session association together.

| At a glance | |
| :--- | :--- |
| **Model** | Managed, first-party event warehouse |
| **Replaces** | Hosted product-analytics tooling |
| **Storage** | Actioneer-operated; first-party, owned by you |
| **Transport** | HTTPS / JSON |
| **Base URL** | `https://api.actioneer.com` |
| **Audience** | Client / data engineering |
| **Auth** | `X-API-Key` header (server-side secret) |

## Before you build

<Warning>
  The API key is a server-side secret. Issue requests from a backend or trusted server context — do not embed it in distributable client binaries or web bundles. Never place secrets, credentials, tokens, or cardholder data inside event or profile properties.
</Warning>

<Info>
  The contract is intentionally analytics-shaped: events, identify calls, and user properties. Wiring up an existing SDK is mostly a change of transport and identity handling, not a re-modeling of your events.
</Info>
