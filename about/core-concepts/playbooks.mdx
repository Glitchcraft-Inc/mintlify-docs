# Playbooks

A Playbook is a validated analytical workflow — a sequence of queries, analysis steps, and output formats — that has been frozen so it can be rerun reliably with fresh data. Playbooks turn one-off analysis into repeatable process.

---

## Why Playbooks exist

Ad-hoc analysis is not reproducible. The same question asked on two different days can produce different SQL, different filters, and different conclusions. If an analyst leaves or a metric definition shifts, the methodology is lost.

Playbooks solve this by capturing the full methodology behind an analysis — not just the final answer, but every step, validation check, and output format — so it can be executed consistently, by anyone, at any time.

---

## How a Playbook is created

Playbooks are typically born from a validated Chat conversation:

<Steps>
  <Step title="Run the analysis in Chat">
    Ask your question, refine the results, confirm the methodology is correct.
  </Step>
  <Step title="Save as Playbook">
    When you are satisfied with the analysis, save it as a Playbook. Actioneer extracts the steps automatically — queries, intermediate logic, output format, and insight generation.
  </Step>
  <Step title="Review and adjust">
    Edit parameters, adjust branching logic, or add validation checks. Playbooks support conditional paths — if a metric crosses a threshold, the analysis can take a different route.
  </Step>
</Steps>

---

## What a Playbook contains

| Component | Description |
| :--- | :--- |
| **Steps** | Ordered sequence of SQL queries, analysis operations, and validation checks |
| **Parameters** | Configurable inputs — date ranges, segments, thresholds — that can change between runs |
| **Branching logic** | Conditional paths that execute based on intermediate results |
| **Output format** | The structure of the final deliverable — charts, tables, narrative summary |
| **Schema binding** | Tied to a specific schema version. If the schema changes, Actioneer auto-adapts or flags the mismatch |

---

## Running a Playbook

Playbooks can be invoked in three ways:

1. **On demand** — Run it manually from the Playbooks page or reference it in Chat.
2. **On schedule** — Set a cadence (daily, weekly, every six hours) and Actioneer runs it automatically, delivering the output to a Board or Slack channel.
3. **Via Scouts** — Automated agents that run Playbooks in the background, grade results, and flag anomalies before anyone has to ask.

<Info>
  Scouts are Actioneer's scheduled monitoring agents. They execute Playbooks on a recurring basis and deliver graded reports — A through E — per title, region, or team. Anomalies are surfaced with root-cause context attached.
</Info>

---

## Schema drift handling

When your data warehouse schema changes — a column is renamed, a table is restructured — Playbooks do not silently break. Actioneer detects the mismatch, attempts to auto-adapt the affected queries, and creates a new versioned copy. If auto-adaptation fails, the Playbook is flagged for manual review.

---

## Playbook discovery

As the number of Playbooks grows, discovery becomes important. Before creating a new Playbook, Actioneer checks whether a similar one already exists (using semantic matching at an 80%+ similarity threshold) and suggests reusing it instead of duplicating effort.
