Every request returns either an acceptance acknowledgement or a single error object. **Errors are a flat `{ "error": "<message>" }` — there is no per-record failure array.**

## Success

```json
{ "status": 1 }
```

The envelope — or every envelope in a `/batch` — passed validation and was queued. Acceptance is not a delivery guarantee; see [Reliability](/ingestion/reliability).

## Status codes

| Code | Meaning | Body | Retry? |
| :--- | :--- | :--- | :--- |
| `200` | Accepted | `{ "status": 1 }` | — |
| `400` | Validation error (e.g. missing `distinct_id`) | `{ "error": "<message>" }` | No — fix the payload |
| `401` | Missing, invalid, unknown, revoked, or inactive API key | `{ "error": "Invalid or unknown api_key" }` | No — fix the key |
| `503` | API-key resolution temporarily unavailable | `{ "error": "api_key resolution temporarily unavailable" }` | Yes — backoff |
| `503` | `/batch` publish temporarily unavailable | `{ "error": "event publish temporarily unavailable" }` | Yes — backoff |

<Note>
  Codes `403`, `413`, `422`, and `429` are not part of the current contract. Validation failures come back as `400`, auth failures as `401`.
</Note>

## Common validation errors

| Message | Cause |
| :--- | :--- |
| `distinct_id is required` | `distinct_id` missing, `null`, or blank |
| `Invalid or unknown api_key` | `X-API-Key` missing or not recognized |

## Handling errors

- **`400`** — a client bug. Don't retry; correct the envelope.
- **`401`** — fix or rotate the key; don't retry as-is.
- **`503`** — transient. Retry with exponential backoff and jitter, reusing the same `event_id` so any already-queued records dedupe. See [Reliability](/ingestion/reliability).

<Warning>
  On a single-event request, `{ "status": 1 }` is returned **before** the asynchronous publish is confirmed — a rare publish failure after acceptance is logged server-side, not surfaced in the response. For critical pipelines, reconcile event counts downstream.
</Warning>
