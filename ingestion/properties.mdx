`properties` is a single object carrying three distinct kinds of value. **They are not separate event types and may coexist in one envelope.**

```text
Describes the action   â†’ place directly in properties         (event property)
Describes the user     â†’ place under $set / $set_once / $unset (profile property)
Identifies the session â†’ send $session_id in properties        (session property)
```

## Event properties

Values scoped to the single event â€” order, plan, screen, campaign, or experiment context; numeric measures (order value, quantity, discount); and request context (URL, referrer, source, medium). **Event properties are never persisted to the user profile.**

Keep transaction figures (e.g. `amount`, `currency`) as event properties â€” they describe the transaction, not the user.

## Profile properties

Profile properties describe the person and persist against the `distinct_id` for later segmentation. They must be wrapped in an operator.

| Operator | Semantics | Idempotent | Example |
| :--- | :--- | :--- | :--- |
| `$set` | Create or overwrite profile fields. | Yes (last-write-wins) | `{ "isPremiumUser": true }` |
| `$set_once` | Write only if the field is absent. | Yes | `{ "first_seen_source": "google" }` |
| `$unset` | Remove fields from the profile. | Yes | `["temporary_flag"]` |

A user-level attribute placed only in event properties will **not** appear on the profile. If an attribute should be queryable per user (e.g. premium status via `isPremiumUser`), send it under `$set`. Pick the operator by whether the value can change:

- `$set` for fields that change â€” plan, role, premium status, and feature flags. Last write wins, so the profile stays current.
- `$set_once` for fixed fields that must never be overwritten â€” `first_seen_source`, signup date, referral code.

<Note>
  **Some context auto-syncs to the profile.** A defined set of context properties â€” `$browser`, `$os`, `$device_type`, `$current_url`, `$referring_domain`, the `utm_*` family, `gclid`, `fbclid`, and more (see [Reserved properties](#reserved-properties)) â€” is copied to the profile automatically when a mutation is processed, without wrapping them in `$set`. Everything else stays event-scoped unless you wrap it in an operator.
</Note>

## Reserved properties

Actioneer reserves a set of names â€” don't use these for your own data.

- **Operators** (wrap profile data): `$set`, `$set_once`, `$unset`, and `$anon_distinct_id` on `$identify`.
- **Special event names:** `$identify` and `$alias` carry identity-merge behavior â€” see [Identity Model](/ingestion/identity).
- **Auto-synced context** (copied to the profile during a mutation): `$browser`, `$browser_version`, `$os`, `$os_version`, `$device_type`, `$app_build`, `$app_name`, `$app_version`, `$app_namespace`, `$current_url`, `$referring_domain`, `$screen_height`, `$screen_width`, `utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content`, `gclid`, `fbclid`, `msclkid`.
- **SDK / session metadata** (event context only, not profile operators): `$lib`, `$lib_version`, `$session_id`, `$device_id`.

<Note>
  Event and property names are **case-sensitive** â€” `Order Completed` and `order completed` are different events.
</Note>

## Ingestion order

`$set`, `$set_once`, and `$unset` resolve in **processing order â€” not by event `timestamp`.** A late, retried, batched, or forwarded event can overwrite a newer `$set` or wrongly satisfy a `$set_once`, because the server stamps each profile update with its own processing time. Within a single event, operators apply as `$set_once` â†’ `$set` â†’ `$unset` â†’ auto-sync.

<Warning>
  Watch this when backfilling history, draining offline mobile queues, or forwarding from a batched pipeline â€” out-of-order arrival can leave a profile holding a stale value. For fields that must never regress, prefer `$set_once`.
</Warning>

## Session property

`$session_id` groups events into a session. Reuse the same value across all events in a session; allocate a new value when a new session begins. The session identifier always travels inside `properties`.

## Combined envelope

This single event records what happened, updates the profile, and attaches the event to a session:

```json
{
  "event_id": "evt-order-001",
  "event": "Order Completed",
  "distinct_id": "user_8a31f0c2",
  "timestamp": "2026-06-04T06:31:00.000Z",
  "properties": {
    "amount": 499,
    "plan": "pro",
    "$session_id": "session-123",
    "$set": { "isPremiumUser": true },
    "$set_once": { "first_seen_source": "google" }
  }
}
```
